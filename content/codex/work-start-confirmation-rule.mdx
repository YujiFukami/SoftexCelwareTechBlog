## はじめに

CodexのようなAIエージェントは、明確な指示があれば作業を速く進められます。一方で、対象範囲や公開有無が曖昧なまま進むと、不要なファイルを編集したり、確認が必要な素材を記事化したりするリスクがあります。

作業開始前に短く進め方を共有しておくと、手戻りを減らせます。

## 使う場面

- 記事追加やMDX変換を始める前
- ツール作成や既存コード修正に入る前
- 候補調査、公開作業、ダッシュボード更新を行う前
- 複数エージェントが同じフォルダで作業しているとき
- 公開リスクや編集範囲の確認が必要なとき

## 確認する内容

作業開始時は、次の5点を短く共有します。

```text
これから次の方針で進めます。

1. 対象: ...
2. 目的: ...
3. 作業内容: ...
4. 公開・匿名化の扱い: ...
5. 完了時の出力: ...
```

毎回長い計画を書く必要はありません。ユーザーが「どこを触り、何を作り、何を報告するのか」を把握できる粒度で十分です。

## 運用のポイント

対象フォルダ、対象記事、対象ファイル、公開有無、リスク確認方針を明示します。

たとえば、MDX記事追加なら「原本Markdownを読み、公開用MDXを新規作成し、顧客名や実パスを出さず、最後に変更ファイルと不足用語を報告する」と伝えます。

既にユーザーが明確に範囲を指定している場合は、確認で止まりすぎず、方針だけ短く出して作業へ進みます。

## 注意点

- 編集範囲が指定されている場合は、その外を触らない
- build、commit、pushの可否を勝手に変更しない
- 機密情報や顧客名が含まれそうな素材は、記事化前に確認する
- 作業が大きい場合は、台帳や進行メモの扱いも確認する

## まとめ

作業開始前の確認は、細かい許可取りではなく、作業の向きをそろえるための短い合図です。

対象、目的、作業内容、公開リスク、完了時の出力を最初に共有しておくと、Codexの作業を複数エージェント環境でも扱いやすくなります。
